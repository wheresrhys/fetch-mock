## Commands

```bash
# Build all packages
npm run build

# Build a single package
npm run build -w packages/fetch-mock

# Lint (auto-fixes)
npm run lint

# Type-check
npm run types:check

# Run tests (Vitest, interactive UI) — excludes jest-specific tests
npm test

# Run tests in CI mode (no UI, with coverage and junit output)
npm run test:ci

# Run tests in browser (Chromium via Playwright)
npm run test:browser

# Run Jest-specific package tests
npm run test:jest

# Run a single test file
npx vitest packages/fetch-mock/src/__tests__/FetchMock/routing.test.js

# Run import compatibility checks
npm run compat:module -w import-compat
```

Commits must follow [Conventional Commits](https://www.conventionalcommits.org/) — enforced by commitlint (`commitlint.config.js`).

## Monorepo Structure

This is an npm workspaces monorepo. The main workspaces are:

- **`packages/fetch-mock`** — the core library (`fetch-mock` npm package)
- **`packages/jest`** — Jest integration (`@fetch-mock/jest`)
- **`packages/vitest`** — Vitest integration (`@fetch-mock/vitest`)
- **`packages/codemods`** — jscodeshift codemods for migrating from older versions (`@fetch-mock/codemods`)
- **`docs/`** — documentation site (separate workspace)
- **`import-compat/`** — compatibility smoke-tests for ESM/CJS import scenarios

Note that type-checking will typically fail before `npm run build` is run at least once.

## Core Architecture (`packages/fetch-mock/src/`)

The library intercepts calls to `fetch` by replacing it with `FetchMock.fetchHandler`. The key classes are:

- **`FetchMock`** — the main entry point and public API. Holds a `Router` and a `CallHistory`. Provides `.mockGlobal()` / `.unmockGlobal()` to swap `globalThis.fetch`, `.spy()` to pass through to real fetch while recording calls, and `.createInstance()` to create isolated instances. Shorthand methods (`get`, `post`, `getOnce`, `any`, `sticky`, etc.) are generated by `defineShorthand` / `defineGreedyShorthand` and delegate to `.route()`.

- **`Router`** — manages the ordered list of `Route` objects. On each fetch call it walks the routes until one matches, then delegates response generation to that route. A fallback route can be set via `.catch()`.

- **`Route`** — represents a single registered route. Composes a list of `MatcherDefinition` functions (built from `UserRouteConfig` fields) into a single predicate, and holds the response config. Supports `repeat`, `delay`, `sticky`, and `waitFor` options.

- **`Matchers`** — exports `builtInMatchers` (an array of `MatcherDefinition`s covering url, method, headers, query, body, params, etc.) and URL-matching helpers. URL matching supports strings, `RegExp`, `URL`, glob patterns, Express-style path params, and `URLMatcherObject` (with keys like `begin`, `end`, `include`, `host`, `path`, `express`, `regexp`).

- **`CallHistory`** — records every `CallLog` (url, options, route hit, response, express params, etc.) and exposes filtering helpers (`calls()`, `called()`, `lastCall()`) used by both the core API and the framework-specific assertion matchers.

- **`RequestUtils`** — utilities for normalising fetch arguments into a `CallLog` before routing.

## Framework Packages

`@fetch-mock/jest` and `@fetch-mock/vitest` each:

1. Subclass `FetchMock` with Jest/Vitest lifecycle methods (`mockClear`, `mockReset`, `mockRestore`).
2. Register custom `expect` matchers (`toHaveFetched`, `toHaveLastFetched`, `toHaveFetchedTimes`, etc.) via `jest-extensions.ts` / `vitest-extensions.ts`.
3. Export a singleton instance and a `manageFetchMockGlobally()` helper that wraps the test runner's global reset/clear/restore hooks.

## Build Output

Each package compiles to both ESM (`dist/esm/`) and CJS (`dist/cjs/`), with `package.json` `exports` fields pointing to the correct format. The `scripts/declare-dist-type.js` script writes `dist/esm/package.json` (`{ "type": "module" }`) and `dist/cjs/package.json` (`{ "type": "commonjs" }`) after TypeScript compilation.
